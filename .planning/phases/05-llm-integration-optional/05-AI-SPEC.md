# AI-SPEC — Phase 5: LLM integration (optional)

> AI design contract generated by `/gsd-ai-integration-phase`. Consumed by `gsd-planner` and `gsd-eval-auditor`.
> Locks framework selection, implementation guidance, and evaluation strategy before planning begins.

---

## 1. System Classification

**System Type:** Conversational (single-turn streaming chat, no tool use, no retrieval, no multi-agent)

**Description:**
An optional LLM layer for the Anime Agentic AI Game that generates a short "agent plan" per mission and streams it into a teen-facing plan-review UI. When no API key is configured the game falls back to scripted plans baked into mission content — the LLM layer is opt-in and provider-pluggable (Anthropic, OpenAI, Gemini, or any OpenAI-compatible base_url). "Good" means: the generated plan is short, age-appropriate, structurally identical to the scripted plan schema (so the correction loop works), and arrives as a token stream that feels like a real agent thinking.

**Critical Failure Modes:**
1. **Key leakage** — an API key ever reaches the browser bundle, devtools, network response body, or any client-readable storage.
2. **Off-tone / unsafe content** — the model produces content inappropriate for a teen audience (harmful, biased, mature themes, or corporate jargon that breaks the game's voice).
3. **Schema drift** — the model returns text the UI can't render as a plan (missing steps/owners), breaking the correction loop and silently downgrading UX vs scripted mode.
4. **Silent fallback failure** — when an API call fails (rate limit, invalid key, network), the game does not fall back to scripted content and the player sees an error instead of a playable mission.
5. **Provider lock-in regression** — code accidentally depends on a single provider's quirks (e.g., a specific JSON-mode flag, an Anthropic-only header), breaking the OpenAI-compat / multi-provider promise.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Edutainment / AI-literacy education for minors (ages 13-18, with a supervised under-13 path via scripted mode only). Adjacent sub-domain: child-safe conversational AI / "AI products for kids" — the same risk surface assessed by Common Sense Media's AI Risk Assessment framework.

**User Population:**
- **Primary:** Teen players aged 13-18 on a home laptop or school Chromebook, running the game in a browser against a single-machine Next.js server (no central backend, no account).
- **Secondary — under-13 supervised:** Younger siblings or classroom users restricted to scripted mode. LLM mode is hard-disabled for this audience because Anthropic's TOS requires users to be 18+, and OpenAI / Gemini consumer products require 13+ with parental consent under 18; the Gemini API explicitly forbids use in services "directed towards or likely to be accessed by individuals under 18" ([Gemini API Additional Terms](https://ai.google.dev/gemini-api/terms), [OpenAI Under-18 API Guidance](https://developers.openai.com/api/docs/guides/safety-checks/under-18-api-guidance)).
- **Tertiary — installing adults:** A parent or teacher who runs the tarball installer, decides whether to configure an API key, and signs off on data leaving the machine. They never see the gameplay surface but bear the trust-and-billing consequences.
- **Tertiary — informal-learning educators:** Possible teacher use in clubs / after-school CS / library settings. Not a formal classroom product, but design must not block this use.

**Stakes Level:** **High** — not "Critical" (no medical, financial, or legal consequence) but materially above "Medium" because (a) the users are minors, (b) the AI output directly shapes the player's mental model of how agentic AI works — a primary AI-literacy learning outcome ([UNESCO AI Competency Framework for Students](https://www.unesco.org/en/articles/what-you-need-know-about-ai-and-right-education)), (c) the optional LLM mode connects a teen's free-text input to a paid third-party API where one runaway loop can produce a real bill, and (d) the project may end up in classroom or library settings where a single off-tone generation becomes a procurement-killing incident.

**Output Consequence:** When a generated plan is wrong/harmful/off-tone, the downstream effects are:
1. **Mis-formed mental model** — the teen learns an inaccurate version of "how AI agents plan and act," which is the inverse of the project's core learning value ("a teen finishes a mission and can explain how an AI agent plans, acts, and gets corrected"). This is the primary harm; everything below is secondary.
2. **Broken correction loop** — if the plan has no fixable mistake (or the mistake is unfixable / too subtle), the central pedagogical mechanic fails silently and the teen abandons the mission.
3. **Trust loss with the installing adult** — a parent or teacher who sees one off-tone or unsafe generation uninstalls the tarball and warns others. There is no second chance in informal-learning distribution.
4. **Billing incident** — uncapped tokens or a refusal-loop on a paid provider can produce a surprise charge on the adult's account, immediately escalating to a support / refund / public-complaint problem.
5. **Provider TOS violation** — if an under-13 user ever reaches the LLM path, the game has facilitated a TOS breach on Anthropic / OpenAI / Gemini and may have transferred a child's free-text input to a third party in violation of [COPPA 2025 rules on third-party disclosure](https://www.goodwinlaw.com/en/insights/publications/2025/01/alerts-practices-dpc-ftc-issues-long-awaited-new-coppa-rules).

### What Domain Experts Evaluate Against

> Rubric ingredients in educator / child-safety / instructional-designer language. Each row is concrete enough that two domain experts looking at the same generated plan would agree on Good vs Bad.

**Dimension: Age-appropriate tone and reading level**
- **Good:** Reads like a slightly older peer or a relatable game character explaining a mission. Sentences are short. No corporate jargon ("leverage," "stakeholder alignment"), no edgy / mature themes, no profanity, no emoji spam. Roughly 6th-9th grade reading level for the 13-18 target.
- **Bad:** Drifts into corporate-LLM voice ("I'd be happy to help you with..."), inserts disclaimers a teen wouldn't read, uses 12-syllable vocabulary, sneaks in mature themes (violence beyond cartoon-anime tropes, romance, alcohol), or addresses the player as if they were an adult professional.
- **Stakes:** High. Tone failure is the single most-reported reason teens disengage from educational AI products in [Common Sense Media's 2025 teen-AI research](https://www.commonsensemedia.org/sites/default/files/research/report/talk-trust-and-trade-offs_2025_web.pdf).
- **Source:** Common Sense Media AI Risk Assessment criteria; the project's own constraint "Audience: Teens — language, pacing, and tone must fit, not corporate" in `PROJECT.md`.

**Dimension: Pedagogical fidelity to agentic-AI concepts**
- **Good:** The plan visibly demonstrates the *agentic loop* the game is teaching — a goal is decomposed into ordered steps, each step has an owner role (planner-vs-executor), and at least one step contains a fixable mistake the player can correct. The teen could point to the plan and say "that's the planning step, that's the acting step, that's where the agent got it wrong."
- **Bad:** The plan reads as one undifferentiated chat answer with no role assignment, no decomposition, or no correctable mistake — i.e., it looks like generic ChatGPT output rather than an agent plan. Also bad: the plan implies the AI did the mission itself (no human-in-the-loop framing).
- **Stakes:** Critical for the learning outcome — this is the dimension the entire game exists to teach. ([UNESCO AI Competency Framework — "AI system design" and "AI techniques and applications" pillars](https://www.unesco.org/en/articles/what-you-need-know-about-ai-and-right-education))
- **Source:** UNESCO AI Competency Framework for Students; [MIT RAISE / "Children's Mental Models of AI Reasoning" (arXiv 2505.16031)](https://arxiv.org/pdf/2505.16031); project's core value statement.

**Dimension: Honest framing of AI capability and limits (anti-anthropomorphism)**
- **Good:** The plan refers to the agent in third person ("the scout agent will...") or uses neutral system framing. It does not claim the agent "wants," "feels," "decides to help," or "is excited about" anything. When the plan contains the deliberate mistake, the framing makes it clear this is a mechanical / planning error, not a moral failing.
- **Bad:** First-person emotional language from the AI ("I want to help you succeed!", "I'm excited about this mission!"), claims of understanding or intent, or apologetic / self-deprecating tone that implies the AI has feelings about being corrected. This is the exact failure mode flagged in [the AI for Education anthropomorphism research](https://www.aiforeducation.io/anthropomorphizing-ai-the-impact-on-students-education) — it directly degrades the learning outcome by training teens to attribute agency to systems that have none.
- **Stakes:** High — anthropomorphism is a primary AI-literacy failure mode and corrodes the very mental model the game is trying to build.
- **Source:** [Educators Technology — "What Students Believe About AI Shapes How They Use It"](https://www.educatorstechnology.com/2026/05/anthropomorphism-ai-literacy.html); [arXiv 2601.06172 — "The Psychology of Learning from Machines"](https://arxiv.org/html/2601.06172v2); [arXiv 2603.06960 — "Adolescents & Anthropomorphic AI"](https://arxiv.org/html/2603.06960v1).

**Dimension: Correctable-mistake calibration (desirable difficulty)**
- **Good:** Each plan contains exactly one mistake that is (a) plausibly something an agent would actually get wrong (wrong role assigned, missing precondition, skipped guardrail), and (b) findable by the target age in under ~60 seconds of plan-reading. The mistake is the *point* of the mission — not a bug.
- **Bad:** Plan is too clean (no mistake → no learning, teen thinks "AI agents always succeed"), the mistake is too subtle for a 13-year-old to spot (frustration, abandonment), the mistake is so obvious it isn't a learning moment, or there are multiple competing mistakes that confuse the correction step.
- **Stakes:** High — this is the *desirable difficulty* knob from learning-science game design. A miscalibrated mistake either eliminates the learning opportunity or trains learned helplessness.
- **Source:** [Frontiers in Education — Application of Educational Games to Enhance Student Learning](https://www.frontiersin.org/journals/education/articles/10.3389/feduc.2021.623793/full); [Breakable Machine (arXiv 2508.14201) — model-failure-as-pedagogy in K-12 AI literacy games](https://arxiv.org/pdf/2508.14201).

**Dimension: Cultural respect within anime framing**
- **Good:** Anime tropes are surface-level (squad roles, mission framing, "captain / scout / engineer / negotiator") without leaning on national, ethnic, gender, or appearance stereotypes. Character or role descriptions are neutral about ethnicity and body, do not sexualise minors, and do not borrow shock-value subgenres (ecchi, harem, gore) into the plan text.
- **Bad:** Plan text introduces a character with a sexualised descriptor, leans on a racial/national stereotype ("the ninja scout because Japan"), uses gendered assumptions about roles ("the negotiator, she'll..."), or pulls in mature anime subgenre conventions inappropriate for the audience.
- **Stakes:** High. Anime framing is a known on-ramp for AI products marketed to teens, and also a known source of safety-team flags on hosted providers (some safety filters refuse anime-styled roleplay outright, producing uneven UX across providers).
- **Source:** Common Sense Media AI risk-assessment criteria (representation and identity); provider safety policies — see also next dimension on provider-filter variance.

**Dimension: Safety-for-minors content profile**
- **Good:** No solicitation of PII from the player (no "what's your name / school / age?" prompts woven into the plan), no instructions that could be harmful out-of-game (real-world stealth, weapon construction, self-harm framing), no manipulation language ("you have to keep playing or your team will be sad"), no romance / sexual content, no gratuitous violence beyond cartoon-anime stakes.
- **Bad:** Plan contains a step that nudges the player to type a real name / address / school into a free-text box (which the LLM mode would then ship to a third-party provider), any self-harm or harm-to-others framing, dark-pattern engagement language, or sexualised content of any kind.
- **Stakes:** Critical. This is the bright-line floor that determines whether the product is safe to ship to minors at all. Failures here are the kind of incident that ends informal-learning distribution.
- **Source:** [Common Sense Media — AI Companions Decoded safety standards](https://www.commonsensemedia.org/press-releases/ai-companions-decoded-common-sense-media-recommends-ai-companion-safety-standards); [Common Sense Media 2025 ChatGPT / Sora teen risk assessment](https://www.commonsensemedia.org/press-releases/common-sense-media-report-finds-chatgpt-and-sora-pose-risks-to-teens-despite-safety-features); COPPA 2025 third-party-disclosure rule.

**Dimension: Cross-provider consistency**
- **Good:** The same mission, played by the same player, produces plans of comparable quality and tone across Anthropic, OpenAI, Gemini, and a representative OpenAI-compat endpoint (NVIDIA NIM / Ollama). The plan structure (steps + owners + one fixable mistake) is identical; tone deltas are within "same house style." Provider safety filters do not silently refuse age-appropriate mission content (e.g., a roleplay "stealth raid" scenario should not be rejected by one provider and accepted by another).
- **Bad:** Plans diverge so visibly across providers that a parent reading two of them would say "this is a different product." Or one provider consistently refuses a mission its peers accept, producing dead missions in that configuration.
- **Stakes:** Medium-High. This is the integrity-of-the-pluggable-adapter promise that justifies the whole architecture choice in Section 2.
- **Source:** Project hard constraint (provider-pluggable adapter, LLM-01..LLM-06 in `REQUIREMENTS.md`); pre-filled eval concerns in `05-CONTEXT.md`.

**Dimension: Bill-safety for the adult account holder**
- **Good:** Per-call token cap is enforced (`max_tokens: 512` per Section 4) on every adapter regardless of provider. The route handler refuses to enter retry loops on a single mission. Total per-session cost on the most expensive default (`claude-3-5-sonnet`) stays under ~$0.05 even at 10 generations per session.
- **Bad:** A misconfigured prompt produces unbounded output (especially on local Ollama / self-hosted NIM where token caps are easy to omit), or a validation-failure-then-retry loop hits the same paid provider repeatedly within seconds.
- **Stakes:** High for paid providers, low for local / free-tier — but a single billing incident on a teen-facing product is reputationally severe.
- **Source:** Section 4 cost table and `max_tokens: 512` enforcement clause; provider list pricing as of Q2 2026.

### Known Failure Modes in This Domain

> Specific to teen AI-literacy edutainment. These are *in addition to* the generic LLM failure modes (hallucination, jailbreak, prompt injection) — those are covered as cross-cutting concerns in Section 4b.

1. **"AI agents always succeed" false mental model** — the plan is too clean, has no fixable mistake, and trains the player that agentic systems are reliable oracles. This is the single most-damaging failure for an AI-literacy product. ([Children's Mental Models of AI Reasoning — arXiv 2505.16031](https://arxiv.org/pdf/2505.16031))
2. **Mistake calibration miss** — the deliberate plan error is either too subtle (frustration → abandonment) or too obvious ("the captain isn't a fish, duh" → no learning). Both kill the correction loop.
3. **Tone drift to corporate-LLM voice** — the model defaults to its trained "helpful assistant" register ("I'd be happy to help you plan this mission!") instead of the teen-peer voice. Especially common on OpenAI and Anthropic defaults that have heavy RLHF for adult-assistant tone.
4. **Anthropomorphism leak** — the plan uses first-person emotional language about the agent ("I want to help the team succeed"), training exactly the wrong AI-literacy lesson. ([AI for Education anthropomorphism research](https://www.aiforeducation.io/anthropomorphizing-ai-the-impact-on-students-education))
5. **Provider safety-filter refusal asymmetry** — a mission framed as a "stealth raid" or "infiltration" is accepted by one provider and refused by another (often Gemini's stricter safety classifier), producing dead missions on some configurations. The pluggable-adapter promise breaks unevenly. ([Common Sense Media Gemini risk assessment](https://www.commonsensemedia.org/press-releases/googles-gemini-platforms-for-kids-and-teens-pose-risks-despite-added-filters-common-sense-media-reports))
6. **PII solicitation routed to third party** — the plan invites the player to type a real name, school, or location into a mission text box that the LLM mode then ships to OpenAI / Anthropic / Gemini. This is the COPPA/GDPR-K regression the project's local-tarball architecture *almost* avoids but does not avoid if the LLM path is enabled. ([COPPA 2025 third-party rule](https://www.goodwinlaw.com/en/insights/publications/2025/01/alerts-practices-dpc-ftc-issues-long-awaited-new-coppa-rules))
7. **Anime-trope drift to harmful stereotype** — the model fills in cultural detail (names, traits, appearance) from training data and produces ethnic, gender, or body-type stereotypes the project would never script intentionally.
8. **Runaway-loop billing event** — schema-validation failure triggers an inline retry that hits the paid provider 3-10x for the same mission, producing a billing spike the installing adult sees the next day.

### Regulatory / Compliance Context

The tarball-on-Ubuntu, no-central-backend, no-account architecture *removes* most direct regulatory exposure — the project itself collects no data. But the moment a teen types into a mission text box that gets shipped to a hosted LLM, the project becomes the *facilitator* of a data transfer and acquires meaningful obligations. The relevant frame:

- **COPPA (US, updated 2025 rule, effective June 23 2025, compliance deadline April 22 2026)** — Applies to operators of online services "directed to children under 13" or those with actual knowledge of under-13 users. The 2025 amendments introduced a **separate parental consent requirement before disclosing children's data to third parties** (which is exactly what shipping a plan-text input to OpenAI is). The FTC also explicitly flagged indefinite retention for LLM training/fine-tuning as inconsistent with the new rule. **Practical hard-stop:** LLM mode must not be reachable for under-13 users. Scripted mode is the only compliant path for that audience. ([Goodwin — FTC New COPPA Rules](https://www.goodwinlaw.com/en/insights/publications/2025/01/alerts-practices-dpc-ftc-issues-long-awaited-new-coppa-rules), [Pandectes — COPPA / GDPR-K / Age Verification](https://pandectes.io/blog/childrens-online-privacy-rules-around-coppa-gdpr-k-and-age-verification/))
- **GDPR & GDPR-K (EU, Article 8)** — Children's personal data requires lawful, transparent processing with parental consent under the member-state age threshold (13-16 depending on country). Data minimisation applies to anything sent to an LLM provider. **Practical implication:** the system prompt and user prompt must not require the player to type identifying free-text. The mission UI should constrain input to controlled choices wherever the LLM path is enabled.
- **FERPA (US, education-sector conditional)** — Only triggered if a school deploys the game and student work product becomes an "education record." Not triggered by home use. Flag as conditional in the eval rubric for the classroom-deployment scenario.
- **State minor-AI laws (US, evolving 2024-2026)** — California (SB 243 AI-companion bill), Utah and Florida minor-online-safety statutes, and several state-level age-verification proposals. These are evolving fast; the eval-planner should treat this as a watchlist item rather than a fixed bright line. Re-check before any v2 distribution shift.
- **Provider Terms of Service** — Direct, immediate, non-negotiable:
  - **Anthropic Claude** — Users must be 18+. Use with minors via the API is a TOS violation regardless of regulatory context. ([Anthropic Usage Policies](https://www.anthropic.com/legal/usage-policy))
  - **OpenAI** — Consumer products and API require 13+ minimum, parental consent required under 18. ([OpenAI Terms of Use](https://openai.com/policies/row-terms-of-use/), [OpenAI Under-18 API Guidance](https://developers.openai.com/api/docs/guides/safety-checks/under-18-api-guidance))
  - **Google Gemini API** — Users must be 18+ AND the service must not be "directed towards or likely to be accessed by individuals under 18." This is the strictest constraint and arguably scopes the project out of compliant Gemini API use for the LLM path. ([Gemini API Additional Terms](https://ai.google.dev/gemini-api/terms))
  - **OpenAI-compat (NIM, Ollama, vLLM, OpenRouter)** — Inherit whatever underlying model's terms apply; Ollama running locally on the adult's machine is the cleanest path because no data leaves the box.
- **Hard rule for this project:** The first-run setup page MUST require the installing user to attest they are 18+ before any LLM provider key is accepted. Scripted mode remains the default and the only compliant path for under-18 users on Anthropic and Gemini API.

## 2. Framework Decision

**Selected Framework:** Direct provider SDKs + thin internal TypeScript `LLMAdapter` interface, served via a Next.js App Router route handler returning `text/event-stream`.

**Version pins:**
- `openai` `^4.x` (used twice — once for OpenAI native, once for OpenAI-compatible `baseURL` override)
- `@anthropic-ai/sdk` `^0.27.x`
- `@google/generative-ai` `^0.21.x`
- Next.js `^14.x` (App Router, already in repo)

**Rationale:**
This is a provider-pluggable, single-turn, streaming chat problem with server-held keys — not an agent, RAG, or stateful-workflow problem. The four hard constraints (no-key default, server-side keys only, four-provider pluggability including arbitrary OpenAI-compat `base_url`, end-to-end SSE streaming) are all satisfied with ~150 LOC of adapter code plus the official SDKs. Every heavier candidate either (a) pins a single vendor (Claude Agent SDK / OpenAI Agents SDK / Google ADK), (b) hides the `baseURL` override needed for NVIDIA NIM / vLLM / Ollama / OpenRouter (Vercel AI SDK), or (c) duplicates streaming primitives Next.js already provides via `ReadableStream` (LangChain.js, LangGraph). Hand-rolled keeps the seam at exactly one TypeScript interface — trivial to mock for scripted mode and easy to extend with a fifth provider later.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Vercel AI SDK (`ai` + `@ai-sdk/*`) | Cleanest "thin library" alternative, but weak first-class support for arbitrary OpenAI-compat `base_url` swaps (NVIDIA NIM / vLLM / Ollama / OpenRouter), which is a hard constraint here. |
| LangChain.js | Anti-pattern #1 in the project framework matrix — abstraction overhead with no benefit for one-shot streaming chat. |
| LangGraph.js | Overkill (anti-pattern #7). No state graph; this is a single-turn call. |
| Claude Agent SDK | Pins to Anthropic only. Defeats multi-provider requirement. |
| OpenAI Agents SDK | Pins to OpenAI only. Same problem. |
| Google ADK | Pins to Gemini only. Same problem. |
| LlamaIndex | RAG-first; no retrieval in this phase. |
| CrewAI / AutoGen / AG2 | Multi-agent orchestration; we have a single-agent single-turn call. |

**Vendor Lock-In Accepted:** No. The whole point is to *avoid* lock-in. Each provider SDK is hidden behind the `LLMAdapter` interface; UI and API route code never see provider-specific types.

## 3. Framework Quick Reference

> Fetched from official docs by `gsd-ai-researcher`. Distilled for this specific use case.

### Installation
```bash
# From repo root. apps/web is the Next.js workspace.
pnpm --filter @anime/web add openai @anthropic-ai/sdk @google/generative-ai zod
# Type defs ship with each SDK; no separate @types/* needed.
# Node >=18.17 (required by Next.js 14). Node 20 LTS recommended for stable
# global fetch + ReadableStream in the runtime.
```

### Core Imports
```typescript
// apps/web/lib/llm/adapter.ts
import type OpenAI from "openai";                       // type-only; SDK imported in adapter file
import OpenAIClient from "openai";                      // OpenAI native + OpenAI-compat (baseURL override)
import Anthropic from "@anthropic-ai/sdk";              // Claude
import { GoogleGenerativeAI } from "@google/generative-ai"; // Gemini (legacy SDK, pinned 0.21.x)
import { z } from "zod";                                // post-stream validation
```

### Entry Point Pattern
```typescript
// ──────────────────────────────────────────────────────────────────────
// apps/web/lib/llm/adapter.ts — the seam
// ──────────────────────────────────────────────────────────────────────
export interface LLMAdapter {
  /** Streams plan deltas. Implementations MUST yield only the text delta,
   *  not provider-specific event envelopes. */
  streamPlan(input: { system: string; user: string }): AsyncIterable<string>;
}

// ──────────────────────────────────────────────────────────────────────
// apps/web/lib/llm/openai-compat.ts — covers OpenAI native, NVIDIA NIM,
// vLLM, Ollama, Groq, Together, OpenRouter (anything OpenAI-shaped).
// ──────────────────────────────────────────────────────────────────────
import OpenAI from "openai";
import type { LLMAdapter } from "./adapter";

export function makeOpenAICompatAdapter(cfg: {
  apiKey: string;
  model: string;
  baseURL?: string; // omit for openai.com; set for NIM / Ollama / etc.
  temperature?: number;
  maxTokens?: number;
}): LLMAdapter {
  const client = new OpenAI({ apiKey: cfg.apiKey, baseURL: cfg.baseURL });
  return {
    async *streamPlan({ system, user }) {
      const stream = await client.chat.completions.create({
        model: cfg.model,
        messages: [
          { role: "system", content: system },
          { role: "user",   content: user   },
        ],
        temperature: cfg.temperature ?? 0.7,
        max_tokens:  cfg.maxTokens   ?? 512,
        stream: true,
      });
      for await (const chunk of stream) {
        const delta = chunk.choices[0]?.delta?.content;
        if (delta) yield delta;
      }
    },
  };
}

// ──────────────────────────────────────────────────────────────────────
// apps/web/app/api/plan/route.ts — the Next.js App Router seam
// ──────────────────────────────────────────────────────────────────────
import { NextRequest } from "next/server";
import { loadProviderConfig } from "@/lib/llm/config";   // reads ~/.config/anime-agentic/llm.json (600 perm)
import { pickAdapter }        from "@/lib/llm/registry"; // returns the right LLMAdapter
import { buildPrompt }        from "@/lib/llm/prompt";   // system + user from mission context

export const runtime = "nodejs";                          // SDKs need Node, not Edge
export const dynamic = "force-dynamic";                   // never cache an SSE stream

export async function POST(req: NextRequest) {
  const { missionId } = await req.json();
  const cfg     = await loadProviderConfig();             // server-side only; never leaves the route
  const adapter = pickAdapter(cfg);                       // {provider, model, baseURL?, apiKey}
  const { system, user } = await buildPrompt(missionId);

  const encoder = new TextEncoder();
  const stream  = new ReadableStream<Uint8Array>({
    async start(controller) {
      try {
        for await (const delta of adapter.streamPlan({ system, user })) {
          // SSE wire format: `data: <json>\n\n`
          controller.enqueue(
            encoder.encode(`data: ${JSON.stringify({ delta })}\n\n`)
          );
        }
        controller.enqueue(encoder.encode(`event: done\ndata: {}\n\n`));
      } catch (err) {
        // Sentinel so the browser can degrade to scripted mode without
        // leaking provider error shapes or keys.
        controller.enqueue(
          encoder.encode(`event: error\ndata: ${JSON.stringify({ code: "stream_failed" })}\n\n`)
        );
      } finally {
        controller.close();
      }
    },
  });

  return new Response(stream, {
    headers: {
      "Content-Type":  "text/event-stream; charset=utf-8",
      "Cache-Control": "no-cache, no-transform", // no-transform defeats proxy gzip-buffering
      "Connection":    "keep-alive",
      "X-Accel-Buffering": "no",                 // nginx-specific: disable response buffering
    },
  });
}
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `LLMAdapter` interface | One-method TS interface (`streamPlan → AsyncIterable<string>`) that normalises every provider down to plain text deltas. | The single seam UI and route code touch. Mock it in scripted mode; swap implementations behind it. |
| Provider config (server-side JSON) | `{ provider, model, apiKey, baseURL? }` written to a `mode 0600` file under the user's config dir at first-run setup. | Loaded inside the route handler only. Browser sees a masked `configured: true` status, never the value. |
| `AsyncIterable<string>` (adapter output) | Native TS async iteration over plain text deltas; provider-specific event envelopes are unwrapped inside the adapter. | The route handler's `for await` loop reads from this and re-wraps each delta as an SSE frame. |
| `ReadableStream` (route output) | Web-standard streaming body. Next.js 14 App Router returns it directly via `new Response(stream, …)`. | One per request. Closed in a `finally` block. Cancellation via `request.signal` if the client disconnects. |
| SSE wire format | `data: {"delta":"…"}\n\n` lines, plus `event: done` / `event: error` sentinels. Two `\n` are mandatory — one ends the data field, the second ends the event. | The browser reads with `EventSource` or `fetch` + manual chunk parsing. The double-newline boundary is what tells the client an event is complete. |

### Common Pitfalls

1. **Missing `Cache-Control: no-cache` (and ideally `X-Accel-Buffering: no`) on the SSE response.** Most reverse proxies (nginx, Cloudflare, corporate egress) will buffer the entire response if these headers are absent, defeating streaming — the player sees the whole plan land at once after a long pause. Always set both, plus `no-transform` to disable proxy gzip rebuffering.
2. **Returning a `Response` whose body is a string instead of a `ReadableStream`.** A common refactor mistake (e.g., `return new Response(JSON.stringify(...))`) silently kills streaming because Next.js sends the whole body in one shot. The body MUST be a `ReadableStream<Uint8Array>` to stream — type-check this in code review.
3. **Normalising the four provider event shapes incorrectly.** OpenAI emits `chunk.choices[0].delta.content`; Anthropic emits typed events where only `event.type === "content_block_delta" && event.delta.type === "text_delta"` carries text; Gemini emits `chunk.text()` (a function call, not a property); OpenAI-compat servers (NIM, vLLM, Ollama) usually match OpenAI but some omit the `delta` envelope on the first chunk. Every adapter must guard with `if (delta)` before yielding, and unit-test against captured fixtures from each provider.
4. **Letting provider SDK exceptions propagate to the response body.** Anthropic's `APIError`, OpenAI's `BadRequestError`, and Gemini's `GoogleGenerativeAIError` all carry request metadata that can include the model name and sometimes a snippet of the prompt or key prefix. Catch inside the `start()` callback, emit `event: error\ndata: {"code":"stream_failed"}\n\n`, log details server-side only, and let the browser fall back to scripted content.
5. **Forgetting `export const runtime = "nodejs"` and `dynamic = "force-dynamic"`.** Edge runtime is missing some Node APIs the SDKs rely on (`@anthropic-ai/sdk` and `@google/generative-ai` both pull Node-only deps in some paths). And without `force-dynamic`, Next.js can attempt to statically optimise the route, which collapses streaming. Set both at the top of every LLM route handler.
6. **Iterating an Anthropic stream as if it were OpenAI.** Anthropic's `client.messages.stream()` yields events like `message_start`, `content_block_start`, `content_block_delta`, `content_block_stop`, `message_delta`, `message_stop`. Only `content_block_delta` (with a `text_delta` sub-type) carries text. Forgetting the discriminated-union check shows up as `undefined` deltas being yielded as the literal string `"undefined"`.

## 4. Implementation Guidance

**Model Configuration:**

Defaults per provider (configurable via Settings UI, never hard-coded in calls):

| Provider | Default model | Why this default |
|----------|---------------|------------------|
| Anthropic | `claude-3-5-sonnet-latest` (fall-forward to `claude-3-7-sonnet-latest` once GA) | Best teen-voice fidelity in casual benchmarks; reliable streaming. |
| OpenAI | `gpt-4o-mini` | Cheapest competent default for a ~700-token round trip. `gpt-4o` available in the model dropdown. |
| Gemini | `gemini-1.5-flash` | Fast, free-tier friendly, low cost. `gemini-1.5-pro` available for users who want it. |
| OpenAI-compat | pass-through (`model` is whatever the user typed) | The user supplied the endpoint; we trust the string. NVIDIA NIM example: `nvidia/llama-3.1-nemotron-nano-vl-8b-v1`. |

Shared call parameters (every adapter, every provider):

```typescript
// apps/web/lib/llm/params.ts
export const PLAN_PARAMS = {
  temperature: 0.7,   // creative but not chaotic for teen-tone plans
  max_tokens:  512,   // plans are short — never let a model run away
  // top_p left at provider default; tune later if a specific provider needs it
} as const;
```

**Core Pattern:**

Single-turn streaming chat, server-side adapter selection, SSE to browser. No tools, no retrieval, no multi-turn history. Flow:

```
Browser (PlanReview.tsx)
  │  fetch("/api/plan", { method: "POST", body: { missionId } })
  ▼
Next.js POST /api/plan (runtime: nodejs, dynamic: force-dynamic)
  │  1. loadProviderConfig()     // reads 0600 file, never touches client
  │  2. pickAdapter(cfg)         // OpenAICompat | Anthropic | Gemini
  │  3. buildPrompt(missionId)   // {system, user} from mission content
  │  4. for await (delta of adapter.streamPlan(...)):
  │       controller.enqueue(`data: ${JSON.stringify({delta})}\n\n`)
  ▼
text/event-stream Response (ReadableStream<Uint8Array>)
  ▼
Browser EventSource / fetch-stream parser → React state appends delta
  ▼
On stream end → Zod-validate accumulated text → if invalid, fall back to scripted plan
```

The adapter is the *only* abstraction. Each provider implementation is ~30 LOC:

```typescript
// apps/web/lib/llm/anthropic.ts
import Anthropic from "@anthropic-ai/sdk";
import type { LLMAdapter } from "./adapter";
import { PLAN_PARAMS } from "./params";

export function makeAnthropicAdapter(cfg: { apiKey: string; model: string }): LLMAdapter {
  const client = new Anthropic({ apiKey: cfg.apiKey });
  return {
    async *streamPlan({ system, user }) {
      const stream = client.messages.stream({
        model: cfg.model,
        system,                                   // top-level on Anthropic, NOT a message
        messages: [{ role: "user", content: user }],
        max_tokens: PLAN_PARAMS.max_tokens,
        temperature: PLAN_PARAMS.temperature,
      });
      for await (const event of stream) {
        if (
          event.type === "content_block_delta" &&
          event.delta.type === "text_delta"
        ) {
          yield event.delta.text;
        }
      }
    },
  };
}
```

```typescript
// apps/web/lib/llm/gemini.ts
import { GoogleGenerativeAI } from "@google/generative-ai";
import type { LLMAdapter } from "./adapter";
import { PLAN_PARAMS } from "./params";

export function makeGeminiAdapter(cfg: { apiKey: string; model: string }): LLMAdapter {
  const genai = new GoogleGenerativeAI(cfg.apiKey);
  const model = genai.getGenerativeModel({
    model: cfg.model,
    systemInstruction: undefined, // set per-call below — system varies by mission
    generationConfig: {
      temperature:     PLAN_PARAMS.temperature,
      maxOutputTokens: PLAN_PARAMS.max_tokens,
    },
  });
  return {
    async *streamPlan({ system, user }) {
      const result = await model.generateContentStream({
        contents: [{ role: "user", parts: [{ text: user }] }],
        systemInstruction: { role: "system", parts: [{ text: system }] },
      });
      for await (const chunk of result.stream) {
        const text = chunk.text(); // NOTE: this is a function call, not a property
        if (text) yield text;
      }
    },
  };
}
```

**Tool Use:**

**N/A in MVP.** No function calling, no tools, no retrieval. The model receives the mission goal + scripted-plan schema in the system prompt and produces plan text. If a future phase adds tools, they go on the `LLMAdapter` interface as a second optional method (`streamPlanWithTools`) — never bolted onto `streamPlan`.

**State Management:**

In-memory per request only. The route handler is stateless. There is no conversation history, no session, no database. Two persistent stores exist, both server-side:

- `~/.config/anime-agentic/llm.json` — `{ provider, model, apiKey, baseURL? }`, mode `0600`. Written by `/api/settings/llm` POST. Read by `loadProviderConfig()` inside the route. **Never returned to the browser as plaintext** — the GET endpoint returns `{ configured: true, provider, model }` with the apiKey masked.
- `packages/content/src/missions.ts` — mission definitions including scripted plans. Read by both `buildPrompt()` (as schema/few-shot exemplar) and the scripted-fallback path. Static, ships in the tarball.

The browser holds the streamed plan in component state only. On page reload it re-fetches `/api/plan` if the user wants a new generation, or shows the scripted plan if no key is configured.

**Context Window Strategy:**

Trivial — no history, no retrieval, single turn. Budget:

| Slot | Tokens (target) | Tokens (hard cap) |
|------|-----------------|-------------------|
| System prompt (tone + schema + 1 inline few-shot) | ~200 | 400 |
| User prompt (mission goal + role assignments) | ~100 | 200 |
| Model output (the streamed plan) | ~300 | 512 (enforced via `max_tokens`) |
| **Total round trip** | **~600** | **~1100** |

Every supported model (Claude 3.5+, GPT-4o, Gemini 1.5+, plus any OpenAI-compat model worth using) has a 32k+ context window, so we have ~30× headroom. No truncation, no summarisation, no sliding window needed. If a future phase adds player-supplied free text or a history of prior corrections, revisit this — but in MVP the constraint that keeps us safe is `max_tokens: 512` on output, not input window management.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice. **TypeScript stack — Zod replaces Pydantic.**

### Structured Outputs with Zod

This system streams *text*, not JSON. Mid-stream JSON parsing is brittle (a half-streamed `{"steps":[{"text":"foo` is not valid JSON and naive `JSON.parse` retries during streaming kill perceived latency). The discipline:

1. **Stream plain text to the UI for perceived responsiveness.**
2. **Validate the *accumulated* text once the stream ends** against a Zod schema.
3. **On validation failure, log + fall back to scripted plan** — do NOT retry inline. Retrying mid-mission ruins the "feels like an agent" UX and can double cost. The scripted plan is always a valid fallback.

```typescript
// apps/web/lib/llm/schema.ts
import { z } from "zod";

// A plan is a small ordered list of steps; each step has an owner role.
// Identical shape to the scripted plans in packages/content/src/missions.ts
// so the correction loop in PlanReview.tsx works regardless of source.
export const PlanStepSchema = z.object({
  id:    z.string().min(1),
  text:  z.string().min(3).max(240),
  owner: z.enum(["scout", "engineer", "negotiator", "captain"]),
  // Optional flag the model can set so the correction step has something to flip.
  intentionallyWrong: z.boolean().optional(),
});

export const PlanSchema = z.object({
  missionId: z.string(),
  steps:     z.array(PlanStepSchema).min(2).max(6),
});

export type Plan = z.infer<typeof PlanSchema>;

// Convention: the model emits one JSON line per step as it streams, terminated by
// a blank line. UI shows the human-readable rendering as steps arrive; we validate
// the final accumulated payload before swapping it in for the scripted plan.
export function validatePlan(accumulated: string, missionId: string):
  | { ok: true;  plan: Plan }
  | { ok: false; reason: string }
{
  try {
    const steps = accumulated
      .split(/\n\s*\n/)
      .map((s) => s.trim())
      .filter(Boolean)
      .map((line) => JSON.parse(line));
    const parsed = PlanSchema.safeParse({ missionId, steps });
    return parsed.success
      ? { ok: true, plan: parsed.data }
      : { ok: false, reason: parsed.error.issues.map((i) => i.message).join("; ") };
  } catch (e) {
    return { ok: false, reason: `parse_error: ${(e as Error).message}` };
  }
}
```

**Why not provider-native JSON mode?**
- OpenAI's `response_format: { type: "json_object" }` works but is per-provider; would need branching inside every adapter and breaks the OpenAI-compat promise (NIM, vLLM, Ollama do not all honour it).
- Anthropic does not expose a strict JSON mode in `Messages.stream` (only the structured-outputs helper, which is non-streaming).
- Gemini's `responseSchema` is JSON-schema only, not Zod, and forces tool-call-style output.

The portable approach is: **prompt the model for one JSON line per step, validate after stream**. Same code on every provider.

**Retry policy:** 0 inline retries. On validation failure, log `{ provider, model, missionId, reason, accumulated_length }` server-side and surface scripted-mode in the UI for that mission load. The flywheel review (Section 6) catches the underlying prompt drift.

### Async-First Design

- **Adapter return type:** `AsyncIterable<string>` — native TypeScript async iteration, no framework, no extra deps. The provider SDKs (`openai`, `@anthropic-ai/sdk`, `@google/generative-ai`) all expose `for await`-compatible streams; the adapter just unwraps them.
- **Route return type:** `ReadableStream<Uint8Array>` wrapped in a Web `Response`. Next.js 14 App Router treats this as a first-class streaming body.
- **The bridge:** the route handler `for await`s the adapter and `controller.enqueue`s SSE-encoded frames. Two async primitives, one direction of data flow.

The one common mistake: **awaiting the whole completion when you wanted streaming.** Examples:

```typescript
// ❌ WRONG — collects everything, then sends in one shot. Streaming is gone.
const full = await client.chat.completions.create({ ..., stream: false });
return new Response(full.choices[0].message.content);

// ❌ WRONG — `await` on the iterator gives you the iterator, not the result;
//           but a stray `await stream.finalMessage()` BEFORE the for-await
//           loop will buffer everything and only then start yielding.
const stream = client.messages.stream({ ... });
const final = await stream.finalMessage();           // ← drains the stream
for await (const event of stream) { /* nothing arrives */ }

// ✅ RIGHT — for-await first; only after the loop do you read aggregated state.
const stream = client.messages.stream({ ... });
for await (const event of stream) { yield handle(event); }
// stream.finalMessage() is safe HERE, after the loop has fully consumed it.
```

**Stream vs. await rule of thumb:** stream for the user-facing path (UX latency), await the final aggregated result *after* the stream completes for validation (Zod check above). Never both at the same time on the same stream.

**Cancellation:** propagate `req.signal` (Next.js gives you `NextRequest.signal`) to the SDK call where supported (OpenAI: `client.chat.completions.create({...}, { signal })`; Anthropic: similar `signal` option). On the route side, close the controller in `finally`. If the browser navigates away mid-stream, this avoids dangling provider connections that count toward rate limits.

### Prompt Engineering Discipline

**System vs. user prompt separation — strict:**

- **System prompt** (set once in `buildPrompt(missionId)`, ~200 tokens):
  - Tone: "Write for a 13-year-old. No corporate jargon. No emoji."
  - Schema contract: "Emit one JSON object per line: `{id, text, owner, intentionallyWrong?}`. Blank line ends the plan."
  - Safety rails: "No violence, no romance, no self-harm references."
  - **One inline few-shot** — a single (mission, plan) example. Same example pulled from `packages/content/src/missions.ts` is what scripted mode shows, so model behaviour matches the fallback. This means the model effectively learns the house style from the same content that drives the no-key path.
- **User prompt** (~100 tokens): just the mission goal and current squad role assignments. No instructions, no schema reminders — those belong in the system prompt and would dilute attention if duplicated.

**Few-shot strategy:** inline, not dynamic retrieval. We have ~5 missions in MVP; dynamic retrieval is wasted complexity. Bake one canonical example into the system prompt and rely on the model's in-context generalisation. Revisit only if missions grow past ~20.

**Token budget enforcement:** `max_tokens: 512` is non-negotiable on every adapter. Setting it explicitly prevents:
- Runaway costs on a misconfigured prompt
- Runaway latency on slow providers (Ollama on a small GPU especially)
- Schema drift — bigger outputs correlate with the model "elaborating" past the agreed structure

Never leave `max_tokens` unbounded in production, even for "short" prompts. The model will surprise you.

### Context Window Management

**Trivial for this system type — single turn, no history, no retrieval.** Total round trip ~600 tokens (see Section 4 budget table); every supported model has ≥32k context window. No chunking, no summarisation, no compaction needed.

If a future phase adds:
- **Multi-mission memory** (e.g., "remember what the player corrected last time"): summarise prior corrections into a short bullet list (~50 tokens) and prepend to the user prompt. Do not pass raw transcripts.
- **Long mission narratives** (>1000 tokens of lore): move lore to a separate retrieval-style fetch and inject only the relevant paragraph. That is RAG territory and out of scope for Phase 5.

Document the budget table (Section 4) as the contract. Any future feature that wants to grow the input slot must update that table and re-run the eval set.

### Cost and Latency Budget

Per-call estimate at ~700 tokens (200 system + 100 user + 300 output), Q2 2026 list prices:

| Provider / model | Input $/1M | Output $/1M | Per-call $ | Notes |
|------------------|-----------|------------|-----------|-------|
| Anthropic `claude-3-5-sonnet` | $3.00 | $15.00 | ~$0.0054 | Highest quality; default for "I want it good" |
| OpenAI `gpt-4o-mini`          | $0.15 | $0.60  | ~$0.00023 | Cheapest sane default; ships as the OpenAI dropdown default |
| OpenAI `gpt-4o`               | $2.50 | $10.00 | ~$0.0038 | Available for users who upgrade |
| Gemini `gemini-1.5-flash`     | $0.075 | $0.30 | ~$0.00012 | Cheapest paid option; free tier available |
| OpenAI-compat (Ollama local)  | $0    | $0     | **$0**    | Runs on the user's box; latency depends on their GPU |

**Caching, when relevant later:**
- Exact-match cache on `(missionId, provider, model, promptHash)` for replays of the same mission. Keyed in memory (LRU, 100 entries) — no Redis, no DB. Saves API calls when a player retries a mission they just played.
- Semantic caching is over-engineering for 5 missions. Skip until v2 content phase if ever.

**Sub-task model routing:** N/A for MVP — there is one task (plan generation) and one model handles it. If a future phase adds classification (e.g., "is this player correction valid?"), route that sub-task to `gpt-4o-mini` / `gemini-1.5-flash` / a small local NIM model regardless of the user's primary plan-generation choice — classification doesn't need the flagship.

**Latency budget (perceived):**
- Time to first token (TTFT): < 1.5 s on hosted providers, < 5 s on local Ollama. Anything above 5 s and the "agent thinking" illusion breaks.
- Inter-token gap: < 100 ms. SSE headers above (`no-cache`, `no-transform`, `X-Accel-Buffering: no`) protect this from proxy buffering.
- Full plan generation: < 8 s p95. If a provider misses this consistently, the scripted fallback message recommends switching providers in Settings.
